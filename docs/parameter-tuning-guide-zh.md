# Visual Commentary Pipeline 参数调优指南

## 1. 这份指南解决什么问题

这个项目能不能出效果，通常不取决于代码能否跑通，而取决于参数是否适合当前视频类型。

同一套默认参数，在下面几类视频上的表现会明显不同：

- PPT 或课件录屏
- 控制台 / Portal 操作演示
- 鼠标点击频繁但页面结构变化较小的视频
- 页面切换很快的产品 demo

因此参数调优的目标，不是单独把某个数值调到“最优”，而是同时平衡三件事：

1. 分段是否合理
2. 讲解是否自然
3. 音频是否稳定落在时间窗内

## 2. 调优时先抓哪几个核心信号

实际调参时，建议先观察下面四个信号：

1. **片段数量是否异常**
   如果切出来的段非常多，说明分段过碎；如果段很少且每段很长，说明分段过粗。

2. **讲解是否重复**
   如果相邻几段一直在说类似的话，通常表示视觉边界切得不对，或单段时间太短，模型拿不到足够稳定的画面差异。

3. **单段音频是否频繁超时**
   如果很多段都要靠最终 `atempo` 硬压，说明当前片段预算偏紧，或者文本天然偏长。

4. **最终听感是否太赶或太碎**
   如果讲解像不断抢拍，往往是片段太短或 `segment_buffer` 太激进；如果讲解明显拖沓，则可能切段太长。

## 3. 参数之间的关系图

可以把主要参数理解成两组联动：

### 3.1 分段控制组

- `--scene-threshold`
- `--min-segment`
- `--max-segment`

这组三个参数共同决定“切成多少段，每段多长”。

### 3.2 时长控制组

- `--segment-buffer`
- `--base-rate`
- `--azure-style`

这组三个参数共同决定“语音多快、可用预算多少、成品听感如何”。

调参时不要只盯着一个参数。例如：

- 你把 `scene-threshold` 调低后，段数会增加。
- 段数增加后，很多段时长会变短。
- 片段变短后，`segment-buffer` 的影响会被放大。
- 预算变紧后，TTS 更容易超时，最终 `atempo` 使用率会上升。

所以参数调优本质上是一个联动过程。

## 4. 各参数的含义、症状与调法

### 4.1 `--scene-threshold`

这是 ffmpeg 场景切换检测阈值，决定系统对画面变化的敏感度。

一般规律：

- 数值更低：更敏感，更容易切出更多段。
- 数值更高：更保守，更不容易切段。

常见症状与处理：

- 如果一个页面里只是鼠标移动、光标闪动、局部区域变化，就被切成很多段：
  说明阈值偏低，应适当调高。

- 如果页面已经明显切换到新模块、新表单或新结果页，但仍被合并在一段里：
  说明阈值偏高，应适当调低。

调参建议：

- PPT / 页面切换明显的视频：可以从 `0.30 ~ 0.38` 起步。
- UI 变化细碎、动画较多的视频：可以尝试 `0.36 ~ 0.45`。
- 页面状态变化小，但需要捕捉细节动作时：可以尝试 `0.24 ~ 0.32`。

### 4.2 `--min-segment`

这个参数控制最短可接受片段。短于该值的片段会被并入前后上下文。

它主要解决的问题是：

- 片段太短，来不及形成自然讲解
- 模型只能输出非常空泛的句子
- TTS 时长稍有波动就会越界

常见症状与处理：

- 如果很多段只有 1 到 2 秒，讲解总在说“这里继续展示……”：
  说明 `min-segment` 太小，应适当调大。

- 如果你希望保留一些短促但有意义的操作动作，却总被并入上一段：
  说明 `min-segment` 可能偏大，应略微调低。

调参建议：

- 稳定的录屏讲解：`3 ~ 4` 秒通常较合适。
- 节奏快的产品 demo：可以试 `2.2 ~ 3.0` 秒。
- 特别碎的页面切换素材：先不要急着继续降 `min-segment`，优先检查 `scene-threshold` 是否过低。

### 4.3 `--max-segment`

这个参数限制单段最长时长，过长片段会被均匀拆开。

它解决的问题是：

- 单段里信息过多，讲解会失焦
- 模型容易写出太长文本
- TTS 时长容易接近上限

常见症状与处理：

- 如果一段里包含“打开页面、填参数、点击部署、查看结果”多个动作，但只生成一句笼统总结：
  通常说明 `max-segment` 太大。

- 如果本来是一页稳定讲解，却被硬拆成多个语义不完整的小段：
  通常说明 `max-segment` 太小。

调参建议：

- 稳定演示视频：`10 ~ 14` 秒通常比较平衡。
- 页面信息量高的视频：可以降到 `8 ~ 10` 秒。
- 节奏较慢、单页面停留更久的视频：可以升到 `12 ~ 16` 秒。

### 4.4 `--segment-buffer`

这个参数为每个片段预留一小段空白时间。真实可用预算是：

$$
\text{usable budget} = \text{segment duration} - \text{segment buffer}
$$

它的作用是给下面这些误差留空间：

- TTS 实际时长波动
- 模型输出长度波动
- 混音与听感上的呼吸感

常见症状与处理：

- 如果音频总是刚好顶到段尾，切换处很挤：
  应适当增大 `segment-buffer`。

- 如果很多短段本来就很紧，却又因为 buffer 太大导致几乎没有可说空间：
  应适当减小 `segment-buffer`。

调参建议：

- 常规推荐值：`0.25 ~ 0.45`
- 片段较长且希望听感从容：`0.40 ~ 0.60`
- 短片段很多的视频：可以尝试 `0.18 ~ 0.30`

### 4.5 `--base-rate`

这是 Azure Speech 的基础语速控制，不直接改文本，只影响口播节奏。

常见症状与处理：

- 如果整体听感偏慢，但文本质量不错：
  可以轻微提高语速，如 `+3%` 或 `+5%`。

- 如果已经频繁需要最终 `atempo`，但不想明显删减内容：
  可以先小幅提高 `base-rate`，再观察是否减少超时。

- 如果语速一快就显得不稳、不自然：
  说明问题不一定在语速，可能在于分段太短。

调参建议：

- 保守起点：`+0%`
- 稍快但仍自然：`+3% ~ +8%`
- 不建议一开始就拉太高，否则容易把文本时长问题伪装成语速问题。

### 4.6 `--azure-style`

这是 Azure Speech 的表达风格参数，不主要影响分段，而主要影响成品的“气质”。

常用思路：

- `professional`：最适合正式演示、教程和方案讲解
- `calm`：更平缓，适合节奏慢、解释型内容
- `cheerful`：更轻快，适合产品展示或营销感更强的 demo

它不是决定是否超时的主参数，但风格变化会影响少量韵律与停顿，因此在边缘预算下也可能带来细微时长差异。

## 5. 推荐使用顺序（先跑 workflow，再调参）

建议先把使用顺序固定下来，再开始细调参数：

1. **先完整跑一次**，生成 manifest、字幕、音轨和成片。
2. **先读 manifest 再判断问题类型**：看是 segment 边界问题、narration 问题，还是 TTS / timing 问题。
3. **如果只有少数片段有问题，优先用 `--resume-from-manifest` + `--segment-id` + `--redo`**，不要整视频重跑。
4. **只有当切段整体不合理时，才用 `--force-replan` 重建 segment plan**。
5. 在这个基础上，再开始调 `scene-threshold`、`min-segment`、`max-segment`、`segment-buffer` 等参数。

这样做的好处是：你先确认问题在哪一层，再决定是重跑单段还是重规划整条视频，成本更低，定位更快。

## 6. 推荐调参顺序

建议按下面顺序调整，而不是同时乱改所有参数：

1. 先调 `scene-threshold`
   目标是让切段数量大致合理，不要过碎也不要过粗。

2. 再调 `min-segment` 和 `max-segment`
   目标是让每段语义相对完整，既能讲清楚，又不会太长。

3. 再调 `segment-buffer`
   目标是让片段边界更稳、听感更自然。

4. 最后再微调 `base-rate` 和 `azure-style`
   目标是让听感和超时率达到更好的平衡。

这个顺序背后的原因是：

- 先把“切段结构”调对
- 再把“时长预算”调稳
- 最后才去修“声音风格”

否则很容易出现用语速问题去掩盖分段问题的情况。

## 6. 不同视频类型的起始参数建议

### 6.1 PPT / Slide 录屏

适合特征：页面切换清晰，每页停留相对稳定。

建议起点：

- `scene-threshold = 0.30 ~ 0.36`
- `min-segment = 3.0 ~ 4.0`
- `max-segment = 10.0 ~ 14.0`
- `segment-buffer = 0.30 ~ 0.45`
- `base-rate = +0%`
- `azure-style = professional`

### 6.2 Azure Portal / 控制台演示

适合特征：局部区域变化较多，面板、表单、弹窗频繁出现。

建议起点：

- `scene-threshold = 0.32 ~ 0.40`
- `min-segment = 2.8 ~ 3.5`
- `max-segment = 8.0 ~ 12.0`
- `segment-buffer = 0.25 ~ 0.40`
- `base-rate = +0% ~ +5%`
- `azure-style = professional`

### 6.3 节奏较快的产品 Demo

适合特征：动作多、页面停留时间短、展示节奏快。

建议起点：

- `scene-threshold = 0.35 ~ 0.45`
- `min-segment = 2.2 ~ 3.0`
- `max-segment = 8.0 ~ 10.0`
- `segment-buffer = 0.18 ~ 0.30`
- `base-rate = +3% ~ +8%`
- `azure-style = professional` 或 `cheerful`

## 7. 常见问题与对策

### 7.1 讲解内容太重复

优先检查：

- `scene-threshold` 是否太低，切出很多视觉差异不大的短段
- `min-segment` 是否太小，导致模型每段都只能说很泛的句子

通常处理方式：

- 略微调高 `scene-threshold`
- 略微调大 `min-segment`

### 7.2 语音经常超时

优先检查：

- `max-segment` 是否太大，单段信息量过多
- `segment-buffer` 是否过大，压缩了有效预算
- `base-rate` 是否过低

通常处理方式：

- 先略降 `max-segment`
- 再评估是否略减 `segment-buffer`
- 最后再考虑小幅提高 `base-rate`

### 7.3 视频里有大量细小动画，切段很乱

优先检查：

- `scene-threshold` 是否过低

通常处理方式：

- 先调高 `scene-threshold`
- 不要第一反应就调大 `min-segment`，否则可能只是把错误切段粗暴合并

### 7.4 听感很赶，但片段看起来已经合理

优先检查：

- `segment-buffer` 是否过大
- `base-rate` 是否过慢
- 是否某类页面本身信息量过高，应该降低 `max-segment`

## 8. 一个实用的调参流程

建议使用同一段 1 到 3 分钟的样本视频反复迭代，而不是每次都跑整条长视频。

推荐流程：

1. 先用默认参数跑一遍样本。
2. 看 manifest 和字幕，判断分段是否合理。
3. 听音频成品，判断是否频繁抢拍或重复。
4. 每次只改 1 到 2 个强相关参数。
5. 记录每轮参数和结果，不要靠记忆比较。

这样做的好处是你能迅速分辨：

- 问题到底在分段
- 还是在时长预算
- 或者只是声音风格不合适

## 9. 建议的默认思路

如果你面对的是一条典型的产品演示录屏，建议先从下面这组思路开始：

- `scene-threshold`: 先求稳，不要过低
- `min-segment`: 保证一句话能自然说完
- `max-segment`: 不让单段语义过重
- `segment-buffer`: 留一点余量，不要卡满
- `base-rate`: 先保持中性
- `azure-style`: 先用 `professional`

这组思路的本质不是绝对数值，而是一个优先级：

> 先保证结构稳定，再追求听感精细化。

## 10. 总结

这套参数没有通吃所有视频的万能答案，但有一个稳定的调优原则：

- 先把切段结构调对
- 再把时长预算调稳
- 最后再修声音风格

如果你在调参时总觉得“怎么改都不对”，大多数时候不是因为单个参数没调准，而是因为你在用后面的参数去弥补前面的结构问题。

对这个项目来说，**正确的分段**永远是第一优先级，后面的讲解质量和 TTS 稳定性都建立在这个基础上。
